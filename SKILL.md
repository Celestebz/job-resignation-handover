# 离职交接

7阶段闭环：岗位选择 → 信息收集 → 资产盘点 → 倒排时间表 → 文档生成 → 未闭环与对外通知 → 最终确认与签收。

核心目标不是"我整理完了"，而是 **接手人能否按这个文件继续工作，并且知道哪些事最急、谁负责、下一步是什么**。适用于任何岗位。

## 核心流程

### 阶段0：岗位选择（优先执行）

先问岗位类型。不同岗位交接内容完全不同，这个决定后续所有表格的字段和结构。

用 AskUserQuestion 单选：

| 选项 | 交接重点 |
|---|---|
| KOL / Influencer Marketing | KOL合作状态、寄样追踪、视频发布、付款、对外通知 |
| 社媒运营 | 账号权限、内容排期、素材库、数据分析 |
| 外贸销售 | 客户跟进、订单状态、报价体系、物流 |
| 客服 / Support | 未处理工单、FAQ、SOP、客户反馈 |
| 产品经理 | PRD、需求排期、干系人、数据埋点 |
| 设计 / 内容 | 源文件、设计规范、版权、外包对接 |
| 开发 / 技术 | 代码仓库、部署流程、环境变量、技术债、线上服务 |
| 其他 | 自定义工作模块，按用户描述生成对应表格 |

**根据岗位生成不同的工作交接看板（tab2）：**

| 岗位 | Tab2 表格 |
|---|---|
| **KOL/Influencer** | ①工作交接总览 ②KOL合作详情（名称/平台/粉丝/产品型号/寄样/tracking/视频发布时间/hold/付款/发票/风险/邮件关键词/下一步） ③待跟进KOL ④客户跟进 ⑤项目交接 |
| **社媒运营** | ①工作交接总览 ②账号清单（平台/账号/粉丝/内容排期/素材库位置/广告投放/数据分析） ③内容日历（已排期未发布） ④项目交接 |
| **外贸销售** | ①工作交接总览 ②在跟客户（名称/国家/联系人/产品/阶段/下一步/金额） ③执行中订单（客户/订单号/产品/数量/金额/交货期/生产进度/物流） ④报价与价格体系 ⑤项目交接 |
| **客服/Support** | ①工作交接总览 ②未处理工单（工单号/客户/问题/状态/优先级/截止时间） ③FAQ与SOP文档 ④常见问题与解决方案 ⑤项目交接 |
| **产品经理** | ①工作交接总览 ②需求清单（需求/优先级/排期/状态/干系人/文档链接） ③进行中项目（项目/背景/进度/下一步/接手人/风险） ④数据埋点与指标 ⑤项目交接 |
| **设计/内容** | ①工作交接总览 ②进行中设计（项目/源文件位置/版本/状态/下一步/接手人） ③素材库与版权（素材类型/位置/授权情况/到期时间） ④外包对接（设计师/项目/状态/联系方式） ⑤项目交接 |
| **开发/技术** | ①工作交接总览 ②代码仓库（仓库名/分支/未合并PR/部署状态/接手人） ③线上服务（服务名/环境/监控/告警/文档） ④环境变量与配置（key名称/用途/位置/是否已移交） ⑤技术债与已知Bug ⑥项目交接 |
| **其他** | ①工作交接总览 ②自定义模块1（按用户描述） ③自定义模块2 ④项目交接 |

### 阶段1：信息收集

用AskUserQuestion批量问，不要一个个挤。分两组问题。

#### 第一组：基本情况（4个问题）

| 问题 | 选项 | 用途 |
|---|---|---|
| 竞业/保密协议 | 都没签/签过保密/签过竞业/都签了 | 决定资源能否带走 |
| 下份方向 | 同行业/换行业/休息/不透露 | 判断资源可复用性 |
| 自建项目归属 | 公司派做/自己折腾/混合/需谈 | 决定数据归属 |
| SaaS账号性质 | 公司开通/自己付费/不确定 | 决定走后能否继续用 |

还要问**离职日期**，这是倒排表的基础。

#### 第二组：接手人视角（4个问题）

| 问题 | 选项 | 用途 |
|---|---|---|
| 谁接手你的工作 | 具体人名/多人分块/待定 | 生成交接看板的核心 |
| 每块工作分别由谁接 | 逐块确认 | 填入工作交接看板 |
| 主管/最终确认人是谁 | 具体人名 | 签收记录用 |
| 离职后是否允许短期联系 | 允许/不允许/看情况 | 填入交接看板备注 |

**关键**：如果接手人还没定，先提醒用户确认再继续。接手人信息是整个交接看板的基础。

#### 第三组：对外通知（3个问题）

| 问题 | 选项 | 用途 |
|---|---|---|
| 哪些人需要通知离职 | 内部同事/外部合作方/KOL/客户/供应商 | 决定通知清单和邮件模板 |
| 分别用什么语言通知 | 中文/英文 | 决定生成哪些邮件模板 |
| 是否需要CC接手人 | 是/否/看情况 | 填入通知清单 |

**关键**：根据回答只生成用得到的邮件模板，不全部堆上去。

### 阶段2：资产盘点

把所有资产分三类，详见 [references/asset-classification.md](references/asset-classification.md)。

- 🔴 **公司资产** — 必须交接，不能带走
- 🟡 **混合资产** — 需判断边界，最容易出事
- 🟢 **个人资产** — 直接带走，无争议

**风险等级**（非法律意见，必要时向HR/法务确认）：
- 高风险：建议交接并留痕，涉及商业敏感信息
- 中风险：建议脱敏处理，只保留方法论
- 低风险：可作为个人经验保留

**关键判断**：签了保密/竞业 + 同行业跳槽 = 风险最高。

### 阶段3：倒排时间表

根据离职日期计算D-30到D-day（或短周期D-12到D-day）。如果离职日期距今不足15天，用12天短周期模板；否则用30天标准模板。

倒排表已内置在HTML工具箱中，交互式任务清单，支持勾选/编辑/添加，状态高亮（今日/逾期/完成）。

### 阶段4：文档生成

复制 `assets/离职交接模板.html` 到用户工作目录，根据阶段0-2的收集结果替换占位符，**按岗位生成对应的工作交接看板**。

#### 占位符替换规则

| 占位符 | 替换为 |
|---|---|
| `{{USER_INFO}}` | 姓名：XX ｜ 岗位：XX ｜ 离职日期：XX ｜ 协议：XX ｜ 下份方向：XX |
| `{{USER_NAME}}` | 用户姓名 |
| `{{RESIGN_DATE}}` | 离职日期（YYYY-MM-DD） |
| `{{ROLE_TABLES}}` | **按阶段0的岗位对应表，生成tab2的工作交接看板HTML**（替换掉模板中的默认KOL表格） |
| `{{ROLE_SIGNOFF_MODULES}}` | 按岗位定制签收记录模块名（替换模板中默认的"KOL合作详情""客户跟进"等） |
| `{{ROLE_CHECKLIST_ITEMS}}` | **AI根据岗位和用户信息动态生成7模块checklist**。格式：`{ "1": ["item1","item2"], "2": [...] }` JS对象。第2模块（账号权限）按实际账号类型填，不出现无关平台的check项 |
| `{{ROLE_ACCOUNT_ROWS}}` | 权限移交表的账号行HTML。按岗位生成：财务填银行U盾/报税CA/财务系统，KOL填社媒/广告，开发填服务器/代码仓库。保留通用行（邮箱/IM）在模板中不动 |
| `{{ROLE_SECURITY_ROWS}}` | 安全检查表的账号行HTML。同上，按岗位生成对应平台 |
| `{{ROLE_ACCOUNT_NOTE}}` | 权限移交方式说明（如财务：U盾→交还、财务系统→转管理员；KOL：社媒→转管理员不共享密码等） |
| `{{ROLE_SECURITY_NOTE}}` | 安全检查提醒（如财务：网银/报税CA/财务系统；KOL：社媒/广告/后台等，尤其容易遗漏） |
| `{{ROLE_ACCOUNT_TYPES}}` | 公司账号类型描述文本（如财务：银行网银、报税CA、财务系统、企业邮箱；开发：代码仓库、服务器、CI/CD、企业邮箱） |
| `{{ROLE_SCHEDULE_ITEMS}}` | **AI根据阶段0-2收集的所有信息动态生成倒排任务**。不预定义，每个岗位/每人生成不同的清单。用 `{ id: "d-neg-12", date: dayOffset(-12), tasks: [...]` 格式的JS数组 |
| `{{ASSET_YELLOW_ROWS}}` | 混合资产具体行（根据用户项目生成） |
| `{{ASSET_GREEN_ROWS}}` | 个人资产具体行 |
| `{{RISK_REMINDER}}` | 基于用户情况的风险提醒 |
| `{{EMAIL_TEMPLATES}}` | 根据阶段1第三组回答，只填充用得到的邮件模板（见下方4模板定义） |

**生成原则**：
- 模板不含任何岗位硬编码。`{{ROLE_TABLES}}` 是整个tab2的table HTML，根据阶段0的映射表生成
- 表格默认2-3个空行，用户可在浏览器中增删行
- **替换顺序**：先替换 `{{ROLE_TABLES}}`、`{{ROLE_SIGNOFF_MODULES}}`、`{{EMAIL_TEMPLATES}}`、`{{ROLE_SCHEDULE_ITEMS}}`、`{{ROLE_CHECKLIST_ITEMS}}`（它们内部可能含其他占位符），再替换 `{{USER_NAME}}`、`{{RESIGN_DATE}}` 等基础占位符
- **倒排任务动态生成规则（`{{ROLE_SCHEDULE_ITEMS}}`）**：
  - 不预定义任何岗位的任务清单。AI根据阶段0（岗位）和阶段1-2（资产、项目、日期、接手人、通知对象）的所有信息，推理出该岗位的具体任务
  - 格式：JS数组 `[ { id: "d-neg-12", date: dayOffset(-12), tasks: [ { id: genId(), name: "任务名", note: "备注", done: false }, ... ] }, ... ]`
  - 前2天（D-12/D-11）的tasks打 `done: true`（信息收集和资产盘点——这是skill本身已完成的工作）
  - 岗位相关任务要贴合实际：财务填报销/银行/税务，KOL填社媒/寄样/视频，开发填代码/部署/环境变量，销售填客户/订单/报价
  - 通用任务保留：邮件交接、IM权限、设备清理、个人数据导出、签收签字

#### 邮件模板定义（按场景×语言）

根据用户回答选择生成，不全部输出。模板 ID 以 `emailContent` 开头，JS 自动识别。

**模板A：对内中文** — 发给主管/HR/同事
```
[称呼]，

我的最后工作日是 [日期]，之后 [接手人] 将接手我的工作，请保持联系。

感谢这段时间的合作。

[你的名字]
```

**模板B：对外英文-正式** — 发给客户/供应商/B2B
```
Dear [Name],

Please be informed that my last day at [Company] will be [Date]. Going forward, [Colleague Name(s)] will be your new point(s) of contact.

Thank you for your partnership.

Sincerely,
[Your Name]
```

**模板C：对外英文-KOL** — 发给KOL/influencer
```
Hey [Name]!

Just a heads-up — my last day at [Company] is [Date]. [Colleague Name(s)] will be taking over and will reach out soon.

Thank you for the great work together — hope our paths cross again!

Cheers,
[Your Name]
```

**模板D：对外中文** — 发给国内合作方
```
[称呼] 您好，

本人将于 [日期] 从 [公司] 离职，后续工作由 [接手人] 对接。

感谢一直以来的合作与支持。

[你的名字]
```

### 阶段5：未闭环事项与对外通知

这两个阶段在生成HTML后，引导用户填写：

1. **未闭环事项**：列出所有尚未闭环的事项——等待回复、等待产品更新、待付款、待发布、待确认等。每项填明：事项、当前状态、风险等级、下一步动作、负责人、截止日期、相关链接、备注。

2. **对外通知**：根据阶段1第三组回答，确认哪些人需要通知，填写通知清单（合作方、关系类型、通知方式、语言、是否CC接手人、通知状态）。邮件模板按场景×语言分4种（对内中文/对外英文正式/对外英文KOL/对外中文），阶段4生成时只输出用得到的。

### 阶段6：最终确认与签收

1. **7大模块Checklist**：工作交接、账号权限、项目归属、个人数据、设备清理、保密协议、离职手续。逐项打勾，有进度条。

2. **逐模块签收记录**：不是只在最后签个字，而是每块工作都有接收记录。字段：模块、交接人、接手人、确认时间、确认方式、是否有遗留问题。

3. **签字确认表**：离职人、直属上级、HR。

4. **交接风险评级**：自动计算低/中/高风险，显示在tab6顶部。
   - 🟢 低风险：所有高优先级事项已交接，账号权限已移交，未闭环事项有负责人
   - 🟡 中风险：还有部分事项未确认接收，或存在未移交账号
   - 🔴 高风险：存在未移交账号/未付款/未通知外部合作方/无接手人

## HTML工具箱结构

单文件6个tab：

1. **倒排表** — 交互式任务清单，支持勾选/编辑/添加/新建分组，状态高亮（今日/逾期/完成），顶部统计+进度条。**默认任务按岗位生成**，不出现KOL等岗位特定词汇。
2. **工作交接看板** — 由 `{{ROLE_TABLES}}` 动态填充。工作交接总览（优先级/紧急程度/接手人/资料位置/口头交接/确认接收）+ **按阶段0岗位映射表生成的岗位特定表格** + 项目交接。所有表格支持增删行。
3. **资产与权限盘点** — 红/黄/绿三类资产 + 账号权限移交清单 + 账号安全检查清单（2FA/recovery email/个人设备/管理员/API key等）
4. **未闭环事项** — 事项/当前状态/风险等级/下一步动作/负责人/截止日期/相关链接/备注。支持增删行。
5. **对外通知** — 通知清单（含语言列）+ 按需生成的邮件模板（4种场景×语言，可编辑可复制）
6. **最终确认/签收** — 风险评级条 + 7模块checklist（含岗位特定项）+ 逐模块签收记录（模块名按岗位替换）+ 签字确认表

所有可编辑表格支持**添加行/删除行**，动态行数存入localStorage。
数据存localStorage（STORAGE_KEY: `resignation-toolkit-v3`），支持JSON导出备份/导入恢复。

## 关键设计决策

1. **接手人视角是核心** — 交接不是"我的清单"，而是"交给谁、什么时候交、对方是否确认"
2. **岗位分流** — 先问岗位再生成，不同岗位的交接看板完全不同。KOL看产品寄样和视频，开发看代码仓库和部署，PM看需求和排期
3. **HTML而非md** — 倒排表是跨几周的todo，需要状态跟踪和勾选，静态md做不到
4. **单文件6 tab** — 方便使用和分享
5. **全通用模板** — 不含任何岗位特定硬编码，通过 `{{ROLE_TABLES}}` 占位符按岗位动态填充
6. **不写入长期memory** — 离职交接信息敏感度极高，默认只在本次会话和生成的HTML/JSON文件中使用
7. **风险等级而非法律判断** — skill不像律师下结论，而是给出风险等级建议，必要时建议向HR/法务确认
8. **表格增删行** — 所有可编辑表格支持动态添加/删除行，数据持久化到localStorage

## 隐私与安全

- **默认不写入长期memory**：离职交接信息包含合同、客户、合作方、账号、付款、公司项目，敏感度很高
- 只在本次会话和生成的HTML/JSON文件中使用
- 如果用户明确要求记住，再保存非敏感偏好
- HTML数据存在浏览器localStorage，不上传任何服务器

## 注意事项

- **阶段0岗位选择必须先执行**，这决定整个tab2的结构
- 倒排表默认任务不出现岗位特定词汇（KOL/1688/TikTok等），用"合作方""项目""账号"等通用描述
- 账号安全检查要逐平台过：根据用户实际管理的平台生成，不预设备份清单。通用项：个人设备/2FA/recovery email/管理员/API key
- 如果用户签了竞业，强烈提醒行业联系人不能带走
- 走之前提醒用户清理个人工具中的公司相关数据
- 模板默认2-3个空行，用户可通过"添加行"按钮扩展
